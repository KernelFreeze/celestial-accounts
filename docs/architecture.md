## 1. Design principles

The service acts as a standalone identity and authorization provider, comparable to Auth0 or Clerk, but self-hosted. Every design decision follows three constraints:

1. **No implicit trust.** Even first-party services authenticate through the same OAuth2 flows as third-party apps. No service gets unscoped access to identity data.
2. **Credentials are pluggable.** Emails, passwords, passkeys, and social logins are all interchangeable "credential" records attached to an account. The account itself is identified by a unique username, never by an email.
3. **OWASP compliance by default.** The OWASP Top Ten and Cheat Sheet Series inform every subsystem, from password hashing parameters to response timing.


## 2. Core identity model

### Account entity

The primary key is an opaque UUID, never exposed in URLs when avoidable. The public-facing identifier is a case-insensitive, validated username. Emails are not identifiers — they are contact methods and can be added or removed. OWASP's authentication guidance recommends allowing usernames separate from email addresses.

### Credential abstraction

Each authentication method is a row in a `credentials` table, linked to an account via `account_id`. The `kind` discriminator determines how the credential is verified:

| kind | credential_data contains | verifier logic |
|---|---|---|
| `password` | Argon2id hash | Constant-time hash comparison |
| `webauthn` | FIDO2 public key + credential ID | Ed25519 challenge-response |
| `oidc` | Provider name + external `sub` claim | OAuth2 code exchange + `id_token` validation |
| `totp` | HMAC secret (encrypted at rest) | RFC 6238 code verification |

An account can hold multiple credentials simultaneously (e.g., a password and a passkey and a Google OIDC link). Removing the last credential of any kind requires re-authentication.

### Database schema (core)

```
accounts
  id              UUID PK
  username        TEXT UNIQUE (lowercased, validated)
  display_name    TEXT
  mfa_enforced    BOOLEAN DEFAULT false
  locked_until    TIMESTAMPTZ NULL
  created_at      TIMESTAMPTZ
  updated_at      TIMESTAMPTZ

credentials
  id              UUID PK
  account_id      UUID FK -> accounts
  kind            TEXT (password | webauthn | oidc | totp)
  provider        TEXT NULL (e.g., "google", "github" for oidc)
  credential_data BYTEA (encrypted at rest)
  verified        BOOLEAN DEFAULT false
  created_at      TIMESTAMPTZ

emails
  id              UUID PK
  account_id      UUID FK -> accounts
  address         TEXT UNIQUE (lowercased)
  verified        BOOLEAN DEFAULT false
  primary         BOOLEAN DEFAULT false
  verified_at     TIMESTAMPTZ NULL
```


## 3. Authentication orchestrator

A single endpoint (`POST /auth/login`) accepts a JSON body with a `kind` discriminator and routes to the matching credential verifier. The orchestrator is responsible for three invariants:

**Uniform response timing.** Every code path — valid account, invalid account, wrong password, locked account — takes the same wall-clock time. When the account doesn't exist, the handler performs a dummy Argon2id hash to equalize latency. OWASP's authentication cheat sheet specifies that responses must return in a consistent amount of time to prevent user enumeration.

**Generic error messages.** The response for a failed login is always `{"error": "invalid_credentials"}` regardless of whether the username was wrong, the password was wrong, or the account is locked. The HTTP status is always `401`.

**Rate limiting before verification.** A per-IP and per-username rate limiter runs before any credential check, preventing brute-force and credential-stuffing attacks.

### 3.1 Password verifier

OWASP's password storage cheat sheet recommends Argon2id as the first choice, with specific parameter guidance:

- **Algorithm:** Argon2id
- **Memory:** 19 MiB minimum (m=19456), 46 MiB recommended (m=47104)
- **Iterations:** t=2 at 19 MiB, t=1 at 46 MiB
- **Parallelism:** p=1
- **Salt:** 16 bytes, generated by the library (Argon2id handles salt internally)
- **Pepper:** A 32-byte secret stored outside the database (in a secrets manager or environment variable), applied as HMAC-SHA-256 pre-hash before Argon2id

Password policy follows OWASP and NIST SP 800-63B:

- Minimum 8 characters.
- Maximum 128 characters (the Argon2id implementation should handle this; no silent truncation).
- All Unicode characters permitted, including whitespace.
- No composition rules (no mandatory uppercase, digits, or symbols).
- No periodic rotation requirements.
- Passwords checked against a breach corpus (e.g., the Have I Been Pwned k-anonymity API) at registration and change time.
- A password strength meter on the client side to guide users, not enforce arbitrary rules.

Comparison uses a constant-time function. The Rust `argon2` crate's `verify_password` does this by default.

### 3.2 Passkey / WebAuthn verifier

Implements the WebAuthn relying-party protocol (FIDO2). The credential table stores the public key and credential ID. The flow:

1. Client requests a challenge from `GET /auth/webauthn/challenge`.
2. Server generates a random challenge, stores it server-side with a 60-second TTL, and returns it.
3. Client signs the challenge with the user's passkey.
4. Server verifies the signature against the stored public key.

Challenges are single-use and bound to the session.

### 3.3 Social / OIDC verifier

Implements the OAuth2 authorization code flow with PKCE against external identity providers (Google, GitHub, Apple, etc.). The flow:

1. Client redirects to `/auth/oidc/{provider}/authorize`, which constructs the provider's authorization URL with `response_type=code`, a cryptographic `state` nonce (bound to the user's session as CSRF protection), PKCE `code_challenge`, and the requested OIDC scopes (`openid profile email`).
2. The provider redirects back to `/auth/oidc/{provider}/callback` with an authorization code.
3. The server exchanges the code for tokens at the provider's token endpoint, validating the `state` parameter.
4. The server validates the `id_token` signature against the provider's JWKS, checks `iss`, `aud`, `exp`, and `nonce`.
5. The `sub` claim from the `id_token` is matched against the `credentials` table. If a matching credential exists, the account is authenticated. If not, and auto-linking is enabled, a new credential is created and linked to the account (or a new account is created).

The redirect URI is validated against a strict exact-match allowlist — no wildcards, no subdomain matching.


## 4. Multi-factor authentication

MFA is modeled as a second credential on the account. After primary authentication succeeds, the orchestrator checks whether MFA is required (by account-level setting or organization-level policy).

If MFA is required, the orchestrator issues a **partial session** — a short-lived PASETO v4.local token (2-minute TTL) that grants access only to the MFA verification endpoint. This token cannot be used for any other API call. The client must complete the MFA challenge to upgrade it to a full session.

Supported second factors:

- **TOTP** (RFC 6238): 6-digit codes, 30-second window, one-window tolerance for clock skew. The HMAC secret is encrypted at rest with AES-256-GCM. TOTP codes are hashed (SHA-256) before comparison for short-term exposure protection, per OWASP MFA guidance.
- **WebAuthn**: A second passkey registered specifically as an MFA factor.

**Recovery codes:** Generated at MFA enrollment time. 10 codes, 12 alphanumeric characters each, generated from CSPRNG. Each code is stored as an Argon2id hash (with low-cost parameters, since the codes have high entropy). Codes are single-use. Users can regenerate the full set, which invalidates all previous codes.

OWASP specifies that all authentication paths must require MFA if any path does — this includes API access, not just the web UI.


## 5. Session management

### Session lifecycle

After successful authentication (including MFA if required), the service creates a session:

1. Generate a 256-bit random session ID from CSPRNG.
2. Hash the session ID with SHA-256 and store the hash in the `sessions` table alongside `account_id`, IP address, user-agent, creation time, and expiration.
3. Issue two tokens to the client:
   - **Access token:** A PASETO v4.public token, 15-minute TTL.
   - **Refresh token:** An opaque 256-bit random value, stored server-side (hashed), 30-day sliding window, 90-day absolute expiry.

### Why PASETO v4

PASETO v4 replaces JWT to eliminate an entire class of token-handling vulnerabilities. JWT's `alg` header allows attackers to manipulate signature verification (e.g., switching from RS256 to HS256, or setting `alg: none`). PASETO removes this by fixing the algorithm per version.

PASETO v4 specifies exactly two modes:

- **v4.public** — Ed25519 signature. The payload is readable by anyone but tamper-proof. Used for access tokens that resource servers need to verify without calling the auth service.
- **v4.local** — XChaCha20 + BLAKE2b MAC. The payload is encrypted and authenticated. Used for partial session tokens, refresh token wrappers, and any token whose contents should not be visible to the client.

The cryptographic primitives in v4 are: Ed25519 for signatures, XChaCha20 for encryption, and BLAKE2b for key derivation and authentication. There is no algorithm negotiation, no key confusion, and no "none" algorithm.

In Rust, the `rusty_paseto` crate provides a type-driven API for v4.public and v4.local tokens.

### Access token payload (v4.public)

```json
{
  "sub": "acct_7f3a...",
  "aud": "https://api.example.com/profile",
  "scope": "profile:read email:read",
  "org_id": "org_9b2c...",
  "iss": "https://auth.example.com",
  "exp": "2026-04-06T12:00:00+00:00",
  "iat": "2026-04-06T11:45:00+00:00",
  "jti": "tok_unique_id"
}
```

The footer carries the key ID (`kid`) so resource servers know which public key to use for verification, enabling key rotation without downtime.

### Implicit assertions

PASETO v4 supports implicit assertions — data that is authenticated (included in the signature or MAC) but not stored in the token. This is used to bind a token to context that is too sensitive to include in the payload:

- The `account_id` can be used as an implicit assertion to prevent token substitution across accounts.
- The `client_id` can be used to bind a token to the OAuth2 client that requested it.

### Cookie configuration

OWASP session management guidance requires the following cookie attributes:

- `Secure` — cookies sent only over HTTPS.
- `HttpOnly` — cookies inaccessible to JavaScript (for the refresh token cookie).
- `SameSite=Lax` — cookies sent on top-level navigations but not on cross-site subrequests. `Strict` for state-changing endpoints.
- `Path=/auth/token` — the refresh token cookie is scoped to the token endpoint only.
- `Domain` — never set explicitly to avoid sharing cookies with subdomains.

### Session revocation

Sessions can be revoked in three ways:

1. **User-initiated:** The user views active sessions (showing IP, user-agent, approximate location, and creation time) and terminates any of them.
2. **Password change:** All sessions for the account are revoked immediately.
3. **Admin action:** Organization admins can revoke all sessions for a member.

Revocation adds the token's `jti` to a short-lived revocation set (Redis or in-memory, TTL matching the access token's remaining lifetime). Resource servers check this set on every request.


## 6. CSRF protection

For browser-based interactions, the service uses the **synchronizer token pattern** as the primary CSRF defense, with SameSite cookies as defense-in-depth. OWASP recommends that SameSite should not be the sole CSRF mitigation.

For API consumers (SPAs), the service supports the **cookie-to-header pattern**: the server sets a CSRF token in a non-HttpOnly cookie (e.g., `XSRF-TOKEN`), and the SPA reads it and attaches it as a custom header (`X-XSRF-TOKEN`) on every mutating request. The server verifies that the header value matches the cookie value.

For pure API clients using Bearer tokens (no cookies), CSRF protection is not needed — the token is not automatically attached by the browser.


## 7. Password reset and account recovery

The flow follows OWASP's forgot password cheat sheet:

1. User submits their username or email to `POST /auth/reset-password/request`.
2. The server always responds with `{"message": "If an account exists, a reset link has been sent."}` — identical body and response time regardless of whether the account exists.
3. If the account exists:
   a. Generate a 256-bit cryptographically random token.
   b. Hash it with SHA-256 and store the hash in a `password_reset_tokens` table with a 20-minute TTL, the `account_id`, and a `used` flag.
   c. Send the plaintext token via email in a reset URL.
4. The reset URL is hardcoded or validated against a trusted-domain allowlist — never derived from the `Host` header, to prevent host header injection.
5. The reset page sets `Referrer-Policy: no-referrer` to prevent token leakage via the `Referer` header.
6. When the user submits the token and a new password to `POST /auth/reset-password/confirm`:
   a. Hash the submitted token, look up the stored hash, verify it exists, is unused, and has not expired.
   b. Validate the new password against the password policy.
   c. Hash the new password with Argon2id.
   d. Store it, mark the token as used, and revoke all existing sessions for the account.
7. Rate limiting is applied per-account and per-IP to the request endpoint.
8. Accounts are never locked in response to reset attempts — OWASP warns that this creates a denial-of-service vector.


## 8. Organizations and RBAC

### Organization model

An account can create or join multiple organizations. Each organization has a URL-safe slug, a display name, and metadata.

```
organizations
  id              UUID PK
  slug            TEXT UNIQUE (lowercased, validated)
  name            TEXT
  mfa_required    BOOLEAN DEFAULT false
  created_at      TIMESTAMPTZ

memberships
  account_id      UUID FK -> accounts
  org_id          UUID FK -> organizations
  role            TEXT (owner | admin | member)
  invited_by      UUID FK -> accounts NULL
  joined_at       TIMESTAMPTZ
  PRIMARY KEY (account_id, org_id)
```

### Roles

Roles are hierarchical: `owner` > `admin` > `member`. Permissions are checked by comparing the requesting account's role against the minimum role required for the action.

- **owner** — full control, including deleting the organization and transferring ownership. At least one owner must exist at all times; the last owner cannot leave or be demoted without transferring ownership first.
- **admin** — invite/remove members, change member roles (up to admin), manage organization settings.
- **member** — read access to organization resources, no management capabilities.

### Invitations

Invitations are token-based links with a 7-day expiry, scoped to a specific email address. The flow:

1. An admin creates an invitation for an email address, specifying a role.
2. The server generates a 256-bit random token, stores its hash alongside the email, org_id, role, and expiry.
3. The email recipient clicks the link. If they have an account, they join the organization. If not, they register first, then join.
4. Accepting an invitation creates a membership record.

```
invitations
  id              UUID PK
  org_id          UUID FK -> organizations
  email           TEXT
  role            TEXT
  token_hash      BYTEA
  invited_by      UUID FK -> accounts
  expires_at      TIMESTAMPTZ
  accepted_at     TIMESTAMPTZ NULL
```

### Authorization enforcement

OWASP's authorization cheat sheet specifies that access control checks must be performed on every request for the specific object or functionality being accessed. The `org_id` and role are carried in the PASETO access token's claims. Each resource server's middleware:

1. Extracts `org_id` from the token.
2. Verifies that the requested resource belongs to that organization.
3. Verifies that the token's role meets the minimum required for the action.

This prevents cross-tenant access even when a valid token is presented.


## 9. OAuth2 authorization server

The auth service is also an OAuth2 authorization server, issuing scoped tokens to both first-party and third-party applications. This ensures that even internal services request only the data they need — the billing service never sees profile data, the notification service never sees billing data.

### Client registry

Every application — first-party or third-party — is registered as a client.

```
clients
  client_id           UUID PK
  client_secret_hash  BYTEA NULL (confidential clients only)
  client_type         TEXT (confidential | public)
  is_first_party      BOOLEAN DEFAULT false
  name                TEXT
  allowed_scopes      TEXT[] (maximum set this client can request)
  redirect_uris       TEXT[] (exact-match, no wildcards)
  consent_skip        BOOLEAN DEFAULT false (only true for 1st-party)
  created_at          TIMESTAMPTZ
```

OWASP's OAuth2 cheat sheet requires that redirect URIs are validated by exact match. Open redirectors must not be exposed.

### Scope taxonomy

Scopes follow a `resource:action` convention. The auth service defines scopes for its own resources:

| Scope | Grants access to |
|---|---|
| `profile:read` | Username, display name, avatar |
| `profile:write` | Update display name, avatar |
| `email:read` | Verified email addresses |
| `sessions:read` | List active sessions |
| `sessions:revoke` | Terminate sessions |
| `org:read` | List organizations and memberships |
| `org:admin` | Invite/remove members, change roles |
| `billing:read` | Read payment methods |
| `billing:write` | Manage payment methods |
| `offline_access` | Issue a refresh token |

Downstream services define their own scopes in the same pattern. A token's `scope` claim contains only the scopes that were requested, approved, and are within the client's `allowed_scopes`.

### Authorization code flow with PKCE

The only interactive grant type. Implicit and resource owner password credentials grants are deprecated per OAuth 2.1 and are not implemented.

1. The client redirects the user to `GET /oauth/authorize` with `response_type=code`, `client_id`, `redirect_uri`, `scope`, `state` (CSRF nonce bound to the user's session), and PKCE `code_challenge` (S256).
2. The auth service authenticates the user if no active session exists (via the orchestrator).
3. **Consent screen.** For third-party clients, the user sees the requested scopes and can approve or deny each one. For first-party clients with `consent_skip = true`, the screen is bypassed — but `allowed_scopes` is still enforced server-side.
4. Consent grants are persisted so users can review and revoke them later.
5. The server issues an authorization code: a 256-bit random value, single-use, 60-second TTL, bound to the `code_challenge`, `client_id`, `redirect_uri`, `account_id`, and approved `scope`.
6. The client exchanges the code at `POST /oauth/token` by presenting the `code_verifier` (and the `client_secret` for confidential clients).
7. The server returns a scoped PASETO v4.public access token and, if `offline_access` was granted, a refresh token.

```
authorization_codes
  code_hash               BYTEA
  client_id               UUID FK -> clients
  account_id              UUID FK -> accounts
  scope                   TEXT
  code_challenge          TEXT
  code_challenge_method   TEXT (S256)
  redirect_uri            TEXT
  expires_at              TIMESTAMPTZ
  used                    BOOLEAN DEFAULT false
```

### Client credentials grant (machine-to-machine)

Backend services that need identity data without a user present authenticate with their `client_id` and `client_secret`. The token is issued with a service-level subject (not a user account), scoped to only the resources that service needs. For example, a notification service gets `email:read` — nothing else.

For machine-to-machine flows, mTLS certificate binding is preferred over bearer tokens. OWASP's OAuth2 guidance recommends sender-constraining mechanisms (DPoP or mTLS) for refresh tokens, and mTLS is the natural fit when there's no browser.

### Audience restriction

Each access token's `aud` claim names the specific resource server the token is valid for. OWASP specifies that access tokens should be restricted to a single resource server. If a client needs to call multiple resource servers, it either:

- Requests separate tokens for each resource server (preferred), or
- Uses token exchange (RFC 8693) to downscope a broader token into audience-specific ones.

Each resource server rejects tokens whose `aud` doesn't match.

### Consent management and revocation

Users can review all authorized applications at `/settings/authorized-apps`:

- Which client has access.
- Which scopes were granted.
- When consent was given.

Revoking consent:

1. Deletes the consent grant record.
2. Invalidates all refresh tokens for that client-account pair.
3. Adds the active access token `jti` values to the revocation set.

```
consent_grants
  account_id      UUID FK -> accounts
  client_id       UUID FK -> clients
  granted_scopes  TEXT[]
  granted_at      TIMESTAMPTZ
  revoked_at      TIMESTAMPTZ NULL
  PRIMARY KEY (account_id, client_id)

refresh_tokens
  token_hash      BYTEA PK
  client_id       UUID FK -> clients
  account_id      UUID FK -> accounts
  scope           TEXT
  issued_at       TIMESTAMPTZ
  expires_at      TIMESTAMPTZ
  rotated_from    BYTEA NULL
  revoked         BOOLEAN DEFAULT false
```

### Refresh token rotation

For interactive clients, refresh tokens are rotated on every use: each token exchange invalidates the old refresh token and issues a new one. If an already-used refresh token is presented again (indicating theft), all tokens in the rotation chain are invalidated and the user's session is terminated.


## 10. Scope enforcement middleware

Every resource server runs an Axum middleware (tower layer) that enforces token validity on every request. The checks, in order:

1. Extract the `Authorization: Bearer` header.
2. Parse and verify the PASETO v4.public token against the auth service's Ed25519 public key (cached from a JWKS-equivalent key endpoint, rotated periodically).
3. Check `iss` matches the expected auth service.
4. Check `aud` matches this resource server's identifier.
5. Check `exp` has not passed.
6. Check the `jti` against the revocation set.
7. Extract `scope` and verify the endpoint's required scope is present. A `GET /profile` requires `profile:read`. A `PUT /profile` requires `profile:write`. Insufficient scope returns `403 Forbidden` with a response header indicating `insufficient_scope`.
8. Extract `org_id` and verify the requested resource belongs to that organization.

If any check fails, the request is rejected before reaching the handler.


## 11. Anti-enumeration measures

User enumeration is prevented at every public-facing endpoint:

| Endpoint | Measure |
|---|---|
| Login | Generic error, equalized timing (dummy hash on miss) |
| Registration | Always says "check your email," even if username is taken |
| Forgot password | Always says "if an account exists, we sent a link" |
| OIDC callback | No distinguishable behavior between link and create |
| Username check | Not exposed as a standalone endpoint |

For registration, if the username is taken, the server performs the same work (hash computation, email dispatch to the already-registered email notifying them of the attempt) to equalize timing.


## 12. Rate limiting and brute-force protection

Rate limits are enforced at three levels:

| Level | Scope | Purpose |
|---|---|---|
| Per-IP | Broad bucket (e.g., 100 req/min) | Slows credential-stuffing bots |
| Per-account | Tighter bucket (e.g., 10 req/min) | Prevents targeted brute-force |
| Per-endpoint | Varies by cost (e.g., 5 req/min for password reset) | Prevents abuse of expensive operations |

After 5 consecutive failed login attempts for a given account, a progressive lockout kicks in: 1 minute, 5 minutes, 15 minutes, 1 hour (exponential backoff). The forgot-password path remains available during lockout, per OWASP guidance.

CAPTCHA is triggered after 3 failed attempts. It is treated as defense-in-depth, not as a primary control — OWASP notes that CAPTCHA can be bypassed by automated solvers.

Implementation: a sliding-window counter in Redis, keyed by `rate:{ip}:{endpoint}` and `rate:{username}:{endpoint}`.


## 13. Audit logging

Every security-relevant event is written to an append-only audit log:

| Event type | Data recorded |
|---|---|
| `login.success` | account_id, ip, user_agent, credential kind, timestamp |
| `login.failure` | username_attempted (hashed), ip, user_agent, reason, timestamp |
| `mfa.challenge` | account_id, factor type, success/failure, timestamp |
| `password.change` | account_id, ip, timestamp |
| `password.reset.request` | ip, timestamp (no account_id to prevent enumeration) |
| `password.reset.confirm` | account_id, ip, timestamp |
| `session.revoke` | account_id, session_id, revoked_by, timestamp |
| `oauth.authorize` | account_id, client_id, scopes, timestamp |
| `oauth.consent.revoke` | account_id, client_id, timestamp |
| `org.member.invite` | org_id, email, role, invited_by, timestamp |
| `org.member.join` | org_id, account_id, role, timestamp |
| `org.member.remove` | org_id, account_id, removed_by, timestamp |

OWASP recommends that details of recent login attempts (date, time, location) be made visible to the user. The audit log powers this: the user's settings page shows the last N login events for their account.

Failed login events store a hash of the attempted username (not plaintext) to enable pattern detection without leaking valid usernames in the logs.

Anomaly detection can consume this log stream to trigger step-up authentication (e.g., CAPTCHA or MFA re-verification) when unusual patterns appear — repeated failures from a new geolocation, logins from multiple countries within a short window, etc.


## 14. Transport and cryptography

### TLS

All endpoints are HTTPS-only. HSTS headers are set with a minimum `max-age` of one year and `includeSubDomains`. The login page and every subsequent authenticated page must be served over TLS — OWASP's authentication cheat sheet makes this a hard requirement.

### PASETO v4 key management

- **v4.public** (access tokens): Ed25519 keypair. The private key signs tokens; the public key is distributed to resource servers via a key endpoint (analogous to JWKS). Keys are rotated on a schedule (e.g., every 90 days), with old public keys retained for verification during the overlap window.
- **v4.local** (partial session tokens, internal tokens): 256-bit symmetric key stored in the secrets manager. Rotated independently of the public keys.

### Encryption at rest

- Password hashes: Argon2id output stored directly (the hash is the protection).
- TOTP secrets, credential data: AES-256-GCM with a key from the secrets manager.
- Pepper: stored in the secrets manager or as an environment variable, never in the database.
- Audit log entries: stored unencrypted (they contain no raw secrets) but in an append-only, tamper-evident store.

### Parameterized queries

All database access uses parameterized queries using Diesel. No string interpolation in SQL. This is the baseline injection prevention required by OWASP.


## 15. XSS prevention

The auth service's own UI (login page, consent screen, settings) follows OWASP's XSS prevention cheat sheet:

- All user-supplied values are contextually escaped before rendering (HTML entity encoding for HTML context, JavaScript encoding for JS context).
- Content Security Policy headers are set: `default-src 'self'; script-src 'self'; style-src 'self'; frame-ancestors 'none'`.
- The `X-Content-Type-Options: nosniff` header prevents MIME-type sniffing.
- The refresh token cookie is `HttpOnly`, preventing JavaScript access.

## 17. Implementation notes for Rust + Axum

These are preliminary notes for the implementation phase:

- **Token handling:** `rusty_paseto` crate for v4.public and v4.local token creation and verification.
- **Password hashing:** `argon2` crate with the `argon2id` variant.
- **WebAuthn:** `webauthn-rs` crate for relying-party operations.
- **Database:** `diesel` with compile-time checked queries and schemas.
- **Rate limiting:** `governor` crate for in-process rate limiting, backed by Redis for distributed deployments.
- **Middleware:** Axum's `tower::Layer` for composing the scope enforcement, rate limiting, and CSRF validation layers.
- **Key rotation:** A background task (tokio interval) that generates new keypairs, publishes the public key to the key endpoint, and retires old private keys after the overlap window.
- **Secrets:** Read from environment variables or a secrets manager at startup; never logged, never serialized to disk.
